UX/UI Overhaul for CTO Dashboard
**Context for AI:** The core Manager + Builder agent logic is battle-tested and working. However, the current UX is acting like a generic chatbot rather than a structured CTO dashboard. We need to fix the UI constraints, separate chat from actions, and build a clear visual hierarchy for task execution.
Please implement the following architectural UX upgrades:
**Priority 1: Plan → Phase → Task → Job Hierarchy View**
 * **Issue:** The Builder tab currently shows a flat, context-less list of jobs.
 * **Fix:** Build a structured tree/hierarchy view. The UI must clearly display the execution flow: Phase 1 (Infra) -> Task (Setup DB) -> Job (Building.../Merged). The user needs to see the holistic plan and what is currently queued vs. running.
**Priority 2: Dedicated PR Approval & Action UI**
 * **Issue:** Approving plans or PRs relies on the user typing "go" or "yes" into the same single chat box. PR diffs are just raw markdown bubbles.
 * **Fix:** Detach approvals from the chat input. Create a structured **PR Approval Card** (preferably in the Builder/Task tab or as an interactive modal) that includes a proper Diff Viewer and explicit **[Approve]** and **[Reject]** buttons.
**Priority 3: Actionable Builder Logs & Failure States**
 * **Issue:** Running milestone logs are unreadable <pre> dumps. Failed jobs just say "failed" with zero context. Retry attempts are invisible.
 * **Fix:** * Replace raw logs with a clean, step-by-step milestone timeline.
   * When a job fails, add an expandable section showing the exact build/test error output.
   * Explicitly show the retry state in the UI (e.g., Attempt 2/3 - Retrying in 60s).
**Priority 4: Input & State Safety (Quality of Life)**
 * **Context Overload:** Disable the main text input box when the agent is paused, or when waiting for a specific button-click approval.
 * **Global Stop:** Ensure the "Stop" action cancels *both* the Manager's thinking process AND forcefully kills the running Builder job/sandbox.
 * **Agent Identity:** Add visual status indicators (e.g., "Manager is thinking...", "Builder is provisioning sandbox...") so the user knows exactly which agent is holding the baton.
